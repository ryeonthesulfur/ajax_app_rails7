【ユーザー認証の導入】

現在のAjaxAppは、世界中の誰でもアクセスできる状態にあります。
閲覧できるユーザーを制限するためにBasic認証を導入しましょう。

--------------------------------------------------------------------------------------------------------------------------------------------------------
 ＊＊「Basic 認証」と「devise」の違い＊＊

 結論から言うと、「鍵をかける」という目的は同じですが、「鍵の頑丈さ」や「用途」が全く違います。


1. Basic認証（マンションの入り口の共通パスワード）
Basic認証は、Webサイト全体に手っ取り早く鍵をかける**「簡易的な共通パスワード」**のようなものです。
Basic認証とは、HTTP通信の規格に備え付けられている、ユーザー認証の仕組みのことです。
サーバーとの通信が可能なユーザーとパスワードをあらかじめ設定しておき、それに一致したユーザーのみが、Webアプリケーションを利用できるようにします。

・特徴: サイトのURLを開いた瞬間に、ブラウザの上からピョコッと小さな入力画面（ポップアップ）が降りてきて、IDとパスワードを求められます。

・仕組み: データベース（テーブル）を使いません。コードの中に直接（または環境変数で）「IDはコレ、パスワードはコレ」と1つだけ決めておきます。

・用途: **「まだ開発中だから、自分と関係者（スクールのメンターなど）以外には絶対に見せたくない！」**という時に、とりあえず全体にサッと鍵をかけるためによく使われます。

・できないこと: 「誰がログインしたか」を区別することはできません（みんな同じ合言葉で入るため）。


2. Devise（マンションの各部屋の個別キー）
一方、Deviseはユーザー一人一人をしっかり管理するための**「本格的なアカウント管理システム」**です。

・特徴: 専用の「新規登録画面（サインアップ）」や「ログイン画面」があり、ユーザーが自分でパスワードを決めることができます。

・仕組み: データベースに「usersテーブル」を作り、そこに一人一人の情報を保存して管理します。

・用途: TwitterやInstagramのように、**「ユーザーごとにマイページを持たせたり、誰がどの投稿をしたのかを記録したい！」**という時に必須になります。

・できること: 「今はharupocoさんがログインしているな」とアプリ側が認識できるので、「harupocoさんの投稿だけを表示する」といった複雑なことができます。



なぜ今回は「Basic認証」を使うの？
現在のAjaxAppはRenderにデプロイされ、URLを知っていれば「世界中の誰でもアクセスして、メモを投稿できる状態」になっています。 
悪意のあるボットなどにスパム投稿をされて荒らされるのを防ぐため、まずは**「手っ取り早く、アプリ全体に関係者専用の鍵（Basic認証）をかけよう！」**というのが今回のカリキュラムの目的です。


【まとめ】

Basic認証 ＝ データベースを使わない、超簡単な「共通の合言葉」。とりあえず見られたくない時に使う。
Devise ＝ データベースを使う、本格的な「個別のアカウント管理」。ユーザーごとに機能を分けたい時に使う。

    ・Basic認証をRailsアプリケーションへ導入しよう

authenticate_or_request_with_http_basic メソッド

「Railsが最初から用意してくれている、Basic認証をめちゃくちゃ簡単に実装できる魔法のメソッド」**です。

1. リクエストする（request）: 画面を開こうとした人に、あの「IDとパスワードを入力してください」という小さなポップアップ（入力窓）を出して要求する。

2. 認証する（authenticate）: 入力されたIDとパスワードが、あらかじめ決めておいたものと一致するかどうかを判定する。



    ・具体的な書き方
Railsのコントローラー（アプリ全体に鍵をかけるなら「app/controllers/application_controller.rb」）に、以下のように記述します。

1. アクセスを検知 ユーザーがあなたのアプリ（AjaxApp）のURLにアクセスします。

2. ポップアップを表示 before_action :basic_auth によって、画面が表示される前に basic_auth メソッドが呼ばれます。
    すると authenticate_or_request_with_http_basic が働き、画面の前に「IDとパスワードを入力してね」というポップアップを出して一時停止します。

3. 入力された値の受け取り ユーザーがポップアップに文字を入力して「ログイン」を押すと、その入力した文字が |username, password| という変数（ブロック引数）の中にスーッと入ってきます。

4. 答え合わせ username == 'admin' && password == '2222' の部分で答え合わせをします。
    もしユーザーが「admin」と「2222」と入力していたら、この式は「正解（true）」になります。
    正解だと、このメソッドは「よし、通ってヨシ！」と判断し、本来のアプリの画面を表示してくれます。
    
    もし間違えていたら「不正解（false）」となり、もう一度ポップアップを出し直して通してくれません。


app/controllers/application_controller.rb
class ApplicationController < ActionController::Base
  before_action :basic_auth


  private

  def basic_auth
    authenticate_or_request_with_http_basic do |username, password|
      username == 'admin' && password == '2222'
    end
  end
end

    ・Basic認証のコードを改良しよう

現状のコードには次の問題点があります。

＊＊ユーザー名とパスワードがコードに記述されている＊＊
Basic認証をするための、adminというユーザー名と、そのパスワード2222が、コードに記述されています。
つまり、GitHub上の公開リポジトリでソースコードを管理している場合、コードを読める何者かに、不正にBasic認証を突破されてしまいます。

    ＊＊シェル＊＊

対策として、コードに直接ユーザー名とパスワードを記述するのではなく、環境変数を利用する実装に切り替える方法をとる。

シェルは、簡単に言うと「ターミナルとOSを繋ぐ窓口役」のような存在です。
ターミナルから入力されたコマンドを読み取って、OSに対して指示を渡し、結果をターミナルに返して表示や実行などの動作をさせます。
「zsh」と呼ばれるファイルに環境変数を定義します。

    ＊＊ zsh ＊＊
「zsh」はログインシェルと呼ばれるもので、プログラムを実行する時に、ユーザーの要求に一番最初に対応する役割を担います。
隠しファイルなので、特別な設定なしではFinderなどには表示されていません。
環境変数を記載する場所は、設定ファイルである「.zshrc」の中です。
設定ファイルは、「vim」というコマンドを用いて編集します。

    ＊＊ vim ＊＊
「vim」とは、サーバー上で使用できるテキストエディタです。
vimコマンドを用いることで、指定したファイルの編集をターミナルから行うことが可能です。
以下が使用例になります。

vim ~/.zshrc

＊＊わかりやすい例え話＊＊
パソコンの仕組みを**「レストラン」**に例えて、それぞれの役割を分かりやすく解説します！

あなた（ユーザー）： レストランのお客さん
ターミナル（黒い画面）： お客さんが座る「テーブル」
OS（Macの脳みそ）： 実際に料理（処理）を作ってくれる「シェフ」

1. シェル（Shell）＝ 「ウェイター」
お客さん（あなた）がテーブル（ターミナル）で「このコマンドを実行して！」と注文した時、**その注文を聞き取って、
厨房のシェフ（OS）に分かりやすい言葉で伝えてくれる「ウェイター」**のことです。

あなたがターミナルに文字を打ち込んでエンターキーを押すと、裏でこのシェル（ウェイター）が走ってシェフに指示を出し、
出来上がった結果をまた画面に表示してくれます。 人間とパソコンを繋ぐ「窓口（通訳）」の役割をしています。

あなたがターミナル（テーブル）で「ls（ファイル一覧見せて！）」や「rails s（サーバー起動して！）」と注文（コマンド入力）すると、
シェルがその注文をOS（シェフ）に伝えてくれます。そして、OSが実行した結果（ファイル一覧やサーバーのログ）を、またあなたのターミナル（テーブル）まで運んできて見せてくれます。
つまり、あなたの命令をOSに伝え、OSからの結果をあなたに見せる、通訳兼メッセンジャーの役割をしてくれるのがシェルです。


    ＊＊「vim ~/.zshrc」とか打つと、そのコマンドはどこに行くんですか？＊＊
これはとても良い質問です！コマンドが流れていく順番はこうなっています。

あなた: ターミナルに 「vim ~/.zshrc」 と入力してEnterキーを押す。
シェル (zsh): その命令を受け取ります。
シェル (zsh): 命令を解読します。「OK！ vim という名前のプログラム（メモ帳）を、~/.zshrc というファイルを開いた状態で起動すればいいんだな！」と理解します。
OS (Macの脳みそ): シェルから「vim を起動して！」という指示を受け取り、実際に vim プログラムを起動して、あなたの画面に表示します。

つまり、あなたの打ったコマンドは、まずシェル(zsh)に届き、解読されてからOSに渡されているのです。


    ＊＊環境変数ってどれのこと？＊＊
環境変数とは、.zshrc の中にあなたが書き込んだ、以下の行そのもののことです。

    export BASIC_AUTH_USER='admin'
    export BASIC_AUTH_PASSWORD='2222'


    ＊＊各単語の意味＊＊
「export」: これから作る『箱』を、OS全体で使えるように公開（エクスポート）します！
「BASIC_AUTH_USER」: 『BASIC_AUTH_USER』という名前の箱を用意して…
「=」: その箱の中に…
「'admin'」: 『admin』という文字列を入れます！


つまり環境変数とは、OS全体で共有できる「名前付きの箱」のことです。 
この箱を zsh（ウェイター）が記憶してくれるので、Railsアプリは ENV["BASIC_AUTH_USER"] という書き方で
「ねえウェイターさん、BASIC_AUTH_USER の箱の中身、教えて！」と尋ねることができるのです。

2. zsh（ゼットシェル）＝ 「zsh君という名前のウェイター」
「シェル」というのは職業の名前で、実際には色々な名前のウェイターがいます。
その中の**「zsh君」という特定のウェイターの名前**です。

一昔前のMacでは「bash（バッシュ）君」というウェイターがメインで働いていたのですが、
最近のMacではより気が利いて優秀な「zsh君」が標準のウェイターとして雇われています。

【今回のカリキュラムとの関係】 
zsh君は、朝出勤した時（ターミナルを開いた時）に、
**.zshrc（ゼットシェル・アールシー）という自分専用の「ルールブック（設定ファイル）」**を必ず読み込みます。 
今回は、このルールブックの中に「Basic認証のパスワードはコレだよ」と秘密のメモ（環境変数）を書き込んで、zsh君に覚えてもらう作業を行います。

＊＊ zsh はファイルなんですか？　それともコマンドのやり取りの橋渡し役なんですか？＊＊
これも素晴らしい質問で、混乱しやすいポイントです！答えは**「両方の側面がある」**ですが、正確にはこうです。

橋渡し役のプログラムとしての 「zsh」: 今まさにあなたのターミナルで動いている**「ウェイター」プログラム本体**の名前が zsh です。これがコマンドのやり取りをしています。

設定ファイルとしての 「.zshrc」: そのウェイタープログラム zsh が、起動時に読み込む**「自分専用のルールブック（設定ファイル）」**の名前が .zshrc です。

zsh（ウェイター）は、毎朝出勤するとき（ターミナルを開くとき）に、必ず .zshrc（ルールブック）を読んで、「今日の秘密の合言葉はこれだな」と記憶します。

3. vim（ヴィム）＝ 「ターミナル専用の特殊なメモ帳」
先ほど出てきた zsh君のルールブック（.zshrc）を開いて文字を書き込むための、
**ターミナルの中で動く「メモ帳アプリ（テキストエディタ）」**のことです。

VSCodeのようにマウスを使ってクリックしたりすることができず、すべてキーボードのコマンドだけで操作するという少し変わった特徴があります。
最初は「文字が入力できない！」「どうやって閉じるの！？」とパニックになりやすい（エンジニアあるあるです！）ですが、カリキュラムの通りにキーボードを叩けば大丈夫です。

vimには「通常モード」と「インサートモード」があります。
通常モードは、コマンドを打つことでファイルを保存したりvimを終了したりできます。
「通常モード」のコマンドには以下のようなものがあります。


コマンド	説明
:w	作成・編集したファイルを保存します。
:q	vimを終了します。
:q!	編集した内容を保存しないでvimを強制終了します。
:wq	編集した内容を保存してvimを強制終了します。


インサートモードは、ファイルに編集を加えることができます。
この場合、「i」キーを押すことで「insert（インサート）モード」になり、文字の入力が可能です。
ファイル編集後は、Escキーを押すと「通常モード」に戻ります。

ファイル編集後は、sourceコマンドを実行する必要があります。



    ＊＊sourceソースコマンド＊＊
「sourceコマンド」とは、シェルに記述された内容を実行する役割を担います。
今回は、zshファイルに記述された内容を実行します。

「source ~/.zshrc」

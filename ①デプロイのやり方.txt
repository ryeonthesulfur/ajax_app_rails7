    ・デプロイ(Render)用の設定ファイルを追加する

① binフォルダに、render-build.shという名称でファイルを新規作成する。
② 作成したファイルに、以下の記述を追加する。

    bin/render-build.sh
    #!/usr/bin/env bash
    # exit on error 
    set -o errexit

    bundle install
    bundle exec rake assets:precompile
    bundle exec rake assets:clean
    bundle exec rake db:migrate

------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
＊＊「render-build.sh」の中身＊＊

一言でいうと、この bin/render-build.sh というファイルは、**「Render（本番サーバー）に、アプリを動かすための準備を自動でやってもらうための『指示書』」**です。

なぜこの指示書が必要なの？
自分のパソコン（開発環境）でRailsアプリを作っているとき、新しくGemを追加したらターミナルで bundle install を叩いたり、データベースの設計を変えたら rails db:migrate を叩いたりしていましたよね。
ですが、Renderのような本番環境のサーバーには、私たちが直接キーボードを叩いてコマンドを入力することができません。

そこで、「ソースコードを受け取ったら、本番環境を立ち上げる前に、この順番でコマンドを打って準備を整えておいてね！」と、あらかじめ Render に渡しておくメモ書きがこのファイルなんです。


各行の詳しい意味
指示書の中身を1行ずつ翻訳してみます！

＊＊#!/usr/bin/env bash の意味＊＊
「今から書く指示は、『Bash（バッシュ）』っていう書き方（ターミナルのコマンド）で書くからね！」というお約束の宣言です。（※おまじないだと思って大丈夫です）
「このファイルはbash（バッシュ）というプログラムで実行してください」という宣言です（シバンと呼ばれます）。スクリプトファイルの1行目に必ず書くお約束のようなものです。

＊＊set -o errexit の意味＊＊
set -o errexit 安全装置です。「もし、この下の作業の途中で1つでもエラーが起きたら、ムリに先に進まずにそこでストップして！」という指示です。
これがないと、データベースの作成に失敗しているのに無理やりアプリを動かそうとして大惨事になるのを防いでくれます。
「途中でエラーが起きたら、そこで作業を中止してね」という設定です（1行目はただのコメントです）。
たとえば bundle install が失敗したのに、そのままデータベースの更新などに進んでしまうと、不完全な状態でアプリが公開されてしまいます。
それを防ぐための重要な安全装置です。

＊＊bundle install の意味＊＊
bundle install おなじみですね！「Gemfile に書いてある便利なパーツ（Gem）たちを、サーバー上にもインストールしてね！」という指示です。

＊＊bundle exec rake assets:precompile の意味＊＊
bundle exec rake assets:precompile 【重要】本番環境特有のコマンドです。
開発中はCSSや画像、JavaScriptをそのまま読み込んでいますが、
本番環境では「少しでも早く画面を表示させるため」に、これらのファイルをギュッと圧縮して1つにまとめる作業が必要です。それをやってくれるコマンドです。
**CSSやJavaScript、画像などのファイル（アセット）を、本番環境用に最適化（圧縮・結合）**します。これにより、Webサイトの読み込み速度が速くなります。


＊＊bundle exec rake assets:clean の意味＊＊
bundle exec rake assets:clean 「何回もデプロイしていると古い画像やCSSのゴミが溜まっちゃうから、不要になった古いものは掃除して捨ててね！」という指示です。
古くなって使われなくなった過去のアセットファイルを削除します。サーバーのディスク容量を節約するための整理整頓です。


＊＊bundle exec rake db:migrate の意味＊＊
bundle exec rake db:migrate これもおなじみですね！「マイグレーションファイルを見て、サーバー上のデータベースにテーブル（データの入れ物）を作ってね！」という指示です。
**データベースの設計図を最新の状態に更新（マイグレーション）**します。新しく追加したテーブルやカラムなどを、本番のデータベースにも反映させます。

＊＊補足事項＊＊

💡 重要なポイント
このファイルを作っただけではRender上で実行できないことがあります。このファイル自身に**「実行する権限」**を与えておく必要があります。 
作成後、ターミナルで以下のコマンドを実行してから、Gitでコミット＆プッシュしてください。

    chmod +x bin/render-build.sh


そして、Renderの管理画面（Dashboard）でアプリを設定する際、
「Build Command（ビルドコマンド）」の入力欄に ./bin/render-build.sh と指定することで、デプロイのたびにこのファイルが自動で実行されるようになります。

    ・秘密情報管理用ファイルを作成

**「アプリで使うパスワードやAPIキーなどの『秘密の情報』を、安全に管理するための初期設定」**の手順を説明します。

1. 2つのファイルの役割について

通常、Railsで新規アプリケーションを作成すると秘密情報を管理するための以下２つのファイルが作成されます。

    credentials.yml.enc：（金庫） 秘密の情報が暗号化されて保存されている金庫のようなファイルです。
                                暗号化されているため、このファイルだけを見ても中身（パスワードなど）は分かりません。

    master.key：（鍵） 上の金庫を開け閉め（暗号化・復号）するための鍵となるファイルです。


しかし、GitHubからクローンしたAjaxAppにはmaster.keyが含まれていません。


 2. なぜ master.key が無いのか？

GitHubなどにプログラムのコードをアップロードする際、「金庫（.enc）」を上げるのは問題ありませんが、
「金庫の鍵（master.key）」まで一緒にアップロードしてしまうと、誰でも秘密情報を見れてしまい大変危険です。

そのため、Railsでは最初から**「master.key は絶対にGitHubにアップロードしない」**という設定がされています。 
今回あなたはGitHubからアプリ（AjaxApp）を自分のパソコンにダウンロード（クローン）してきたため、金庫はあるけど、それを開けるための鍵が手元にない状態になっています。


3. 手順の詳しい解説

鍵がないと今ある金庫は一生開けられないため、**「古い金庫を一度捨てて、新しい金庫と新しい鍵を作り直す」**という作業を行います。
以下の手順で作成を行いましょう。
まず、以下の古い金庫を削除しましょう。

    config/credentials.yml.enc

続いて、ターミナルから下記のコマンドを入力し新しい金庫と鍵を作成しましょう。

    EDITOR="vi" bin/rails credentials:edit

このコマンドを実行すると、Railsが**「あ、金庫も鍵もないね！じゃあ両方新しく作って、中身を編集できるようにするよ！」**と
自動で新しい credentials.yml.enc と master.key のペアを作成してくれます。

    bin/rails credentials:edit：金庫の中身を編集するコマンドです。  
        
    EDITOR="vi"：「その金庫の中身を『vi（ブイアイ）』というターミナル上のメモ帳（エディタ）で開いてね」という指示です。

この作業を行うことで、あなた専用の新しい鍵と金庫が作成され、本番環境（Renderなど）で安全に秘密情報を扱う準備が整います。



    ・GitHubにコードをプッシュしよう

Renderでデプロイを行う場合、あらかじめGitHubにコードをプッシュしておく必要があります。
GitHub Desktopを使用して、AjaxAppをGitHubにプッシュしましょう。

    ・Renderでデータベースを作成する

まず、ヘッダーに表示されている「+ New」のボタンをクリック
メニューが開くので、「Postgres」を選択します。
作成するデータベースの名称を指定します。ここでは分かりやすいよう「ajaxapp_production」と入力しましょう。
また使用するリージョンを、以下のように「Ohio」に変更します。
この変更は、デフォルトで設定されるOregonで障害の発生率が高いことから、それを回避するためのものです。

この際、必ずInstance TypeをFree（無料）に変更しましょう。

変更できたら、「Create Database」をクリック
データベースの作成が始まり、「Status」に「Creating」と表示されます。作成には少し時間がかかります。

Statusが「Available」に変わったらデータベースの作成は完了


画面を下にスクロールすると「Internal Database URL」という欄があります。

後ほど、ここに記載されているURLが必要になるので、下図のようにアイコンをクリックしてクリップボードにコピーしましょう。

postgresql://ajaxapp_production_57dn_user:Ql2M0QVMluc8BxGX61I3NKXBXMTUYgS9@dpg-d71n7075gffc73fvdr00-a/ajaxapp_production_57dn

コピーした内容は、メモなどに貼り付け、後でコピペできるようにしておきましょう。


    ・Renderでアプリを新規作成

次にRenderでアプリの新規作成を行います。
ヘッダーにある「+ New」のボタンをクリック
メニューの中から「Web Service」を選択

GitHubのリポジトリと連携させるための設定画面が表示されます。

リストの中から「ajax_app_rails7」を選択し、右下にある「Connect」のボタンをクリックしましょう。

Name欄にアプリケーションの名称を入力します。

アプリケーション名は、ajax_app_rails7_user_idとしてください。例えば、テックマスターのユーザーIDが123456であれば、ajax_app_rails7_123456というアプリケーション名になります。

また、Languageは「Ruby」を選択してください。
Regionはデータベース作成時と同様に「Ohio」に変更しましょう。
データベースとアプリのリージョンが異なるとエラーが発生します。リージョンの選択に誤りがないよう注意しましょう。


アプリのデプロイや起動時のコマンドを入力します。
以下２か所の欄を編集します。初期状態で入力されているコマンドを削除して、以下の内容を入力しましょう。

入力欄	入力内容
Build Command	./bin/render-build.sh
Start Command	bundle exec puma -C config/puma.rb



画面をスクロールして「Environment (環境変数)」の追加を行います。
デフォルトで記載されている「WEB_CONCURRENCY」はそのままにしておきましょう。


これから２つの環境変数を設定します。

まず右下の「Edit」をクリックし、編集可能な状態にしてください。
１つ目の設定はマスターキーの設定です。

「RAILS_MASTER_KEY」と入力されている右側に、アプリケーションのマスターキーを入力します。
マスターキーは、ローカルのアプリケーション内にあります。VSCodeでajax_app_rails7を表示させ、configフォルダにある「master.key」ファイルの内容をコピペします。


２つ目は、データベースURLの設定です。
まず、環境変数の入力欄を追加するため、「Add Environment Variable」のボタンをクリックしましょう。


続いて、以下のように２つの欄に入力を行いましょう。

① 左側に「DATABASE_URL」と入力します。
② 右側に、先ほどデータベース作成時にコピーした「Internal Database URL」の内容をコピペします。



以上でajax_app_rails7の環境変数の設定は完了です。
アプリによっては追加の環境変数が必要になりますので、その場合は忘れないように設定を行いましょう。

ここまで入力できたら、画面の最下部にある「Deploy Web Service」のボタンをクリックしましょう。


デプロイが完了すると、緑色のアイコンで「Live」と表示され、
ターミナルに「Puma starting in cluster mode...」という文字が見えます。

行が赤くなり削除対象になっていることがわかりますが、この時点では実際には削除は行われていません。
置き換えのアイコンをクリックして削除完了。


4. 不具合の対応

    エラーログから原因を探ろう

アプリケーションが表示されなかったり、意図した通りに動作しない場合は、原因を探るためエラーログを確認する必要があります。

その場合は、メニューから「Logs」を選択してログを表示させましょう。



    テーブルの変更がRenderに反映しない場合の対応

カラムの追加などのテーブルの変更を行った
ローカルでは、変更内容が正しく反映する
Renderでデプロイを行った際に、エラーは出ないがテーブルに上記の変更が反映されない

以下の作業を行うと、それまでに投稿していたデータは全て削除されます。それでも支障がない場合のみ実行しましょう。

Renderで作成したデータベースをリセットしよう

    ・設定ファイルの内容を変更しよう

#!/usr/bin/env bash
# exit on error
set -o errexit

bundle install
bundle exec rake assets:precompile
bundle exec rake assets:clean
# bundle exec rake db:migrate
DISABLE_DATABASE_ENVIRONMENT_CHECK=1 bundle exec rake db:migrate:reset

DISABLE_DATABASE_ENVIRONMENT_CHECK=1は、本番環境のデータベースを削除するために必要な設定です。
本番用のデータベースには重要なデータが含まれるため、通常は削除できないようになっています。

bundle exec rake db:migrate:resetは、データベースをリセットするためのコマンドです。
以下の３つを順に実行したのと同じ動作をします。

rake db:drop（データベースの削除）
rake db:create（データベースの作成）
rake db:migrate（マイグレーションファイルの実行）


    ・TablePlusを終了しよう
これからリセットしようとしているデータベースをTablePlusで開いている場合、削除することができません。
もしTablePlusを開いている場合は一度終了しましょう。


    ・GitHubにプッシュしよう

    ・動作確認を行おう

    ・設定を元に戻しておこう

今の設定では、デプロイを行うごとに、必ずデータベースがリセットされます。
通常そのような動作は必要ないため、以下のように設定ファイルを元に戻しておきましょう。
8行目のコメントアウトを外して、9行目の記述を削除します。

・データベースを休止状態にするための操作
